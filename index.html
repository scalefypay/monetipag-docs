Monetipag API Documentation

Complete payment gateway with support for PIX and Credit Card.

Base URL

https://api.monetipag.com

Authentication

All API requests must include the authentication header:

HTTP Header 
Authorization: Bearer <token>
Important: The authentication token must be obtained via the admin panel or through the authentication endpoint.

JavaScript SDK (Frontend)

The JavaScript SDK enables secure credit card tokenization directly in the browser, ensuring PCI DSS compliance.

Installation

HTML 
<script type="module">
  import { createCh } from "https://scalefy.sfo3.cdn.digitaloceanspaces.com/sdk/v1/script.min.js";
</script>

Card Tokenization

Use the encrypt() method to tokenize card data before sending it to the API.

Parameters

Field Type Required Description
number string Yes Card number (no spaces)
holder_name string Yes Name printed on the card
exp_month string Yes Expiry month (MM)
exp_year string Yes Expiry year (YYYY)
cvv string Yes Security code (3-4 digits)

Usage Example

JavaScript 
import { createCh } from "https://scalefy.sfo3.cdn.digitaloceanspaces.com/sdk/v1/script.min.js";

const cardData = {
  number: "4111111111111111",
  holder_name: "João Silva",
  exp_month: "12",
  exp_year: "2025",
  cvv: "123"
};

const token = await createCh.encrypt(cardData);
console.log(token); // Use this token when creating the invoice

Response

JSON 
{
  "token": "tok_abc123xyz456..."
}

Cria uma nova fatura para pagamento via PIX ou Cartão de Crédito.

Parâmetros do Body

Nível Raiz

Field Type Required Description

ip_payer string No Payer IP address

postback_url string No URL to receive status change notifications

payer Object

Field Type Required Description

address_number string Yes Address number
address_complement string No Address complement
address_city string Yes City
address_state string Yes State (abbreviation, e.g. SP)

card Object

Field Type Required Description

items Object

Field Type Required Description

description string No Item description

List Invoices

GET /api/v2/invoices

Lists the invoices for the authenticated company, with support for filters and pagination.

Query Parameters

Parameter Type Required Description
page int No Page number (default: 1)
limit int No Items per page (default: 20, max: 100)
start_date string No Start date in YYYY-MM-DD format
end_date string No End date in YYYY-MM-DD format (inclusive up to 23:59:59)
payment_method string No Payment method: pix, credit_card or bank_slip
status string No Invoice status: paid, pending, refused, canceled, chargeback, refunded, etc.

Request Examples

HTTP 
GET https://api.monetipag.com/api/v2/invoices
Authorization: Bearer <token>

# With pagination
GET https://api.monetipag.com/api/v2/invoices?page=1&limit=10

# With date filter
GET https://api.monetipag.com/api/v2/invoices?start_date=2026-01-01&end_date=2026-01-31

# With method and status filter
GET https://api.monetipag.com/api/v2/invoices?payment_method=pix&status=paid

# Combining filters
GET https://api.monetipag.com/api/v2/invoices?start_date=2026-03-01&status=pending&page=2&limit=50

PCI DSS: The client data is sanitized following the same standard as the individual query endpoint. PII fields return [REDACTED_PII] and documents show only the last 4 digits.

Payout (Withdrawal)

POST /api/v2/payout

Requests a PIX withdrawal to the registered account or to a PIX key provided in the request.

Authentication: This endpoint requires the private API Key sent in the Authorization header.

Body Parameters

Field Type Required Description
amount integer Yes Amount in cents (minimum 1000 = R$10.00)
type string Yes Always "withdrawal"
description string No Free-form description
notes string No Additional notes
postback_url string No URL to receive status change notifications
pix_key string No Destination PIX key (if different from the registered one)
pix_key_type string No PIX key type: cpf, cnpj, email, phone, random

Request Example

JSON 
{
  "amount": 5000,
  "type": "withdrawal",
  "description": "Monthly withdrawal",
  "notes": "Optional note",
  "postback_url": "https://yoursite.com/webhook/withdraw",
  "pix_key": "email@example.com",
  "pix_key_type": "email"
}

Response Example (200 OK)

JSON 
{
  "message": "Withdrawal processed successfully",
  "data": {
    "id": 123,
    "status": "processing",
    "amount": 5000,
    "type": "withdrawal",
    "has_baas": true,
    "postback_url": "https://yoursite.com/webhook/withdraw",
    "pix_key": "email@exemplo.com",
    "pix_key_type": "email",
    "can_auto_process": true,
    "balances": {
      "pix": 10000,
      "card": 5000,
      "bank_slip": 0,
      "total": 15000
    }
  }
}

Response Fields

Field Description
id Withdrawal order ID
status pending, processing, completed, failed, rejected
has_baas Whether BaaS is configured for automatic processing
can_auto_process Whether the withdrawal was automatically processed via BaaS
balances Updated balances after the debit (in cents)

Status Postback

When configured, the postback_url will receive a POST request on each withdrawal status change:

JSON 
{
  "id": 123,
  "status": "completed",
  "amount": 5000,
  "type": "withdrawal",
  "end_to_end_id": "E12345678901234567890123",
  "rejection_reason": null
}

Withdrawal Status

Status Description
processing Being processed by BaaS
completed Withdrawal completed successfully
failed Processing failed
rejected Rejected by admin or BaaS

Response Fields

Field Type Description

Get Payout

GET /api/v2/payout/:id

Returns the webhook payload of a payout (withdrawal) for the authenticated user's company.

Authentication: Requires authentication token (userID via Gin context). The user must be associated with a company.

Path Parameters

Parameter Type Description
id string Payout identifier (interpreted according to the by parameter)

Query Parameters

Parameter Type Default Description
by string tid Search type. Possible values: tid (or id), external, transaction, e2e, any

Details of the by parameter

Value Description
tid / id Search by the internal (numeric) ID of the WithdrawalOrder
external Search by external_id
transaction Search by transaction_id
e2e Search by baas_end_to_end_id
any Tries first by numeric ID, then by external_id, transaction_id or baas_end_to_end_id

Request Examples

HTTP 
GET /payout/42?by=tid
GET /payout/abc123?by=external
GET /payout/E00012345?by=e2e
GET /payout/any-value?by=any

Response 200 OK

JSON 
{
  "id": 123,
  "status": "completed",
  "amount": 15000,
  "type": "withdrawal",
  "end_to_end_id": "E123456789",
  "rejection_reason": null
}

Response Fields

Field Type Description
id integer / string Payout ID
status string processing, completed, failed or rejected
amount integer Amount in cents
type string Always "withdrawal"
end_to_end_id string | null Present only when status = "completed"
rejection_reason string | null Present only when status = "failed" or "rejected"
Status normalization: The pending status and any unknown value are normalized to processing.

Possible Errors

Status Body When
401 {"error": "User not authenticated"} Token missing or invalid
403 {"error": "No company found for this user"} User has no associated company
400 {"error": "id must be numeric when by=tid"} by=tid with non-numeric ID
400 {"error": "invalid 'by' param"} Invalid value for the by parameter
404 {"error": "withdrawal not found"} Payout not found for this company
500 {"error": "database error"} Internal database error

Check Balance

GET /api/v2/company/balance

Returns the available balance of the company, broken down by payment method.

Authentication: This endpoint requires the private API Key sent in the Authorization header.

Response Example (200 OK)

JSON 
{
  "message": "Balance retrieved successfully",
  "data": {
    "pix": 8000,
    "card": 5000,
    "bank_slip": 2000,
    "total": 15000
  }
}

Response Fields

Field Description
pix Balance from PIX payments (in cents)
card Balance from card payments (in cents)
bank_slip Balance from bank slip payments (in cents)
total Total available balance (in cents)

Invoice Statuses

Status Description

Error Codes

HTTP codes returned by the API:

Code Description
200 Successful request
400 Invalid request (bad request)
401 Unauthorized (invalid or missing token)
404 Resource not found
422 Validation error in submitted fields
500 Internal server error

Postback (Webhook)

When the postback_url field is set when creating an invoice, the API will send a POST request to that URL whenever the invoice status changes.

Webhook Payload

JSON 
{
  "invoice_id": "inv_123456789",
  "status": "paid",
  "payment_method": "pix",
  "amount": 10000,
  "currency": "BRL",
  "paid_at": "2026-02-11T14:20:00Z"
}
Tip: Your endpoint should respond with HTTP status 200 to confirm receipt. The API will retry the webhook in case of failure.
